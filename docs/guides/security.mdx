# Security Best Practices

This guide covers security configuration and best practices for deploying Briefly CRM in production.

<Callout type="warning">
    Never deploy with debug mode enabled or expose sensitive credentials.
</Callout>

## Environment Security

### Required Environment Variables

Set these in production:

```bash
# Server
NODE_ENV=production
PORT=3000

# Database - use secure connection string
DATABASE_URL=postgresql://user:password@host:5432/database?sslmode=require

# Authentication - use a strong secret (32+ characters)
BETTER_AUTH_SECRET=your-secure-secret-key-min-32-chars-long
BETTER_AUTH_URL=https://your-domain.com
```

### Redis Security

Redis should be configured with authentication:

```bash
REDIS_HOST=localhost
REDIS_PORT=6379
# Use REDIS_PASSWORD if your Redis requires auth
```

### Rate Limiting

Rate limiting is enabled by default:

| Endpoint Type | Limit   | Window |
| ------------- | ------- | ------ |
| Auth          | 10/min  | per IP |
| API           | 100/min | per IP |
| Bulk          | 20/min  | per IP |

### CORS Configuration

Restrict CORS origins in production:

```bash
CORS_ORIGIN=https://your-production-domain.com
```

## Authentication

### Better Auth Configuration

Better Auth provides enterprise-grade authentication with:

- Secure password hashing (bcrypt)
- Session management with secure cookies
- OAuth 2.0 support (Google)
- API token authentication

### Password Requirements

The system enforces:

- Minimum password length: 8 characters
- Hashing algorithm: bcrypt with cost factor 10

### Session Security

Sessions are managed with:

- Secure, HttpOnly cookies
- SameSite policy set to strict
- Automatic expiration (30 days default)

### API Tokens

API tokens provide programmatic access:

```bash
# Include in Authorization header
Authorization: Bearer your_api_token
```

Tokens have the same permissions as the user who created them.

## Role-Based Access Control

### Permission Model

Granular permissions control access to resources:

| Resource       | Actions                   |
| -------------- | ------------------------- |
| `customers`    | `read`, `write`, `delete` |
| `orders`       | `read`, `write`, `delete` |
| `products`     | `read`, `write`, `delete` |
| `imports`      | `read`, `write`           |
| `exports`      | `read`, `write`           |
| `integrations` | `read`, `write`, `delete` |
| `segments`     | `read`, `write`, `delete` |

### Default Roles

| Role     | Description                                     |
| -------- | ----------------------------------------------- |
| `root`   | Full access to all resources and settings       |
| `admin`  | Admin access except organization deletion       |
| `member` | Read-only access to customers, orders, products |

### Custom Roles

Create custom roles with specific permissions:

```http
POST /roles
```

```json
{
    "name": "Sales Team",
    "permissions": {
        "customers": ["read", "write"],
        "orders": ["read"],
        "products": ["read"],
        "segments": ["read"]
    }
}
```

### Organization Scoping

All data is scoped to organizations. Users can only access data within their organization.

## Input Validation

### Request Validation

All requests are validated using Zod schemas:

- Invalid requests return 400 Bad Request
- Validation errors include field-specific messages

### SQL Injection Prevention

Prisma ORM protects against SQL injection through:

- Parameterized queries
- Type-safe data handling

### XSS Prevention

Response data is sanitized where necessary. The API returns JSON which is inherently safe from XSS.

## API Security

### Authentication Required

All API endpoints (except auth) require valid authentication:

```bash
# Invalid token returns 401
Authorization: Bearer invalid_token
# {"message": "Invalid token"}
```

### Authorization Checks

Endpoints check user permissions:

```bash
# Insufficient permissions returns 403
Authorization: Bearer member_token
# {"message": "Insufficient permissions"}
```

## Webhook Security

### Shopify Webhooks

Shopify webhooks are validated using HMAC:

- Header: `X-Shopify-Hmac-Sha256`
- Signature is verified before processing

### Idempotency Keys

Duplicate webhooks are rejected within 24 hours:

```
{idempotency-key} = {integrationId}:{shopifyId}:{topic}
```

## Data Security

### PostgreSQL

Use SSL connections in production:

```bash
DATABASE_URL=postgresql://user:password@host:5432/database?sslmode=require
```

### Redis

For production, use Redis with:

- Password authentication
- TLS encryption
- Allowed IPs configured

### Logging

Sensitive data is not logged:

- Passwords are never logged
- API tokens are masked in logs
- Request bodies containing secrets are redacted

## Monitoring

### Sentry Integration

Set `SENTRY_DSN` to enable error tracking:

```bash
SENTRY_DSN=https://xxx@sentry.io/xxx
```

### Log Retention

Logs are written to stdout in development. In production, configure log aggregation.

## Security Checklist

Before deploying to production:

- [ ] Set `NODE_ENV=production`
- [ ] Use strong `BETTER_AUTH_SECRET` (32+ characters)
- [ ] Configure restrictive CORS origin
- [ ] Use PostgreSQL with SSL
- [ ] Use Redis with authentication
- [ ] Enable rate limiting
- [ ] Configure Sentry DSN for error tracking
- [ ] Review role permissions
- [ ] Use HTTPS for all traffic
- [ ] Configure firewall rules

## Reporting Security Issues

If you discover a security vulnerability, please report it through appropriate channels.
